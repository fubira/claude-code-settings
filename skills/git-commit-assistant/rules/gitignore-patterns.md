## Pattern Matching Rules

### Wildcards

- `*` : Matches any string of characters (except `/`)
- `**` : Matches any string including `/` (directory recursion)
- `?` : Matches any single character
- `[abc]` : Matches any character in the set
- `[!abc]` : Matches any character not in the set

### Examples

```gitignore
# Ignore all .log files
*.log

# Ignore all .log files in any subdirectory
**/*.log

# Ignore specific file
secret.key

# Ignore directory and all contents
node_modules/

# Ignore files matching pattern
temp-*
*-backup
```

### Negation

Use `!` to negate a pattern (include files that would otherwise be ignored):

```gitignore
# Ignore all .log files
*.log

# But include important.log
!important.log

# Ignore everything in build/
build/*

# But keep build/README.md
!build/README.md
```

## Usage Guidelines

### When Adding Patterns

1. **Critical patterns first**
   - Always add credentials/secrets patterns
   - Then MCP configs and personal settings

2. **Tech stack specific**
   - Add patterns for your project's languages/frameworks

3. **Team coordination**
   - Discuss with team before adding broad patterns
   - Document why patterns were added

### When NOT to Add Patterns

- Don't add patterns for files that should be committed
- Don't add patterns that are too broad
- Don't add IDE patterns if team uses different tools (use global .gitignore instead)

### Global vs. Repository .gitignore

**Global** (`~/.gitignore_global`):

- Personal IDE settings
- OS-specific files
- Personal tools

**Repository**:

- Project build artifacts
- Dependencies
- Credentials/secrets
- MCP configurations

## Validation Checklist

Before committing .gitignore changes:

- [ ] Critical patterns present (credentials, MCP configs)
- [ ] No overly broad patterns that might exclude necessary files
- [ ] Patterns tested (verify files are actually ignored)
- [ ] Team-specific patterns discussed
- [ ] Comments added for non-obvious patterns
