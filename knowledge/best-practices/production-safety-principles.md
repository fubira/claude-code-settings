# 本番環境安全性の原則

実運用システムにおける障害を予防・最小化するための原則。

## 1. 出力先の分離

**原則**: 異なる目的のファイルは異なるディレクトリに保存する。1つのディレクトリに複数の形式を混在させない。

**理由**: ビルドツールやフレームワークはディレクトリ単位で glob する。混在すると意図しないファイルが読み込まれ、ビルド失敗や誤動作の原因になる。

**具体例**:
- CDNに同期するディレクトリに分析用ファイルを置かない
- 静的サイトジェネレーターのコンテンツディレクトリに一時ファイルを置かない
- 同じ拡張子で異なるスキーマのファイルを同一ディレクトリに混在させない

## 2. CDN/外部ストレージとの双方向同期の危険性

**原則**: 外部ストレージとの同期がある場合、ローカル削除だけでは不十分。アップロード先にも残っており、次回同期で復活する。

**対策**:
- 不要ファイルのアップロードを防ぐ（書き込み時点で正しいディレクトリに保存）
- アップロードロジックにフィルタを入れる（特定パターンを除外）
- 問題発生時は「ローカル削除 + リモート削除 + キャッシュクリア」の3点セット

## 3. デプロイ前に影響範囲を予測する

**原則**: 新しいファイルを生成する機能を追加する際、そのファイルが他のシステム（ビルドツール、同期、CDN等）にどう影響するか確認する。

**チェックリスト**:
- [ ] そのファイルは glob で拾われるか？（`**/*.json` 等）
- [ ] そのファイルはCDN/外部ストレージに同期されるか？
- [ ] そのファイルはビルドツールに読み込まれるか？
- [ ] 削除する場合、どこから削除が必要か？（ローカル、リモート、キャッシュ）

## 4. 緊急修正の即時適用手段を確保する

**原則**: Dockerビルド+デプロイのパイプラインだけに頼らない。パイプラインが遅い場合のバイパス手段を用意する。

**手段の例**:
- ボリュームマウントされたソースコードの直接編集
- サーバー上での `git pull` + 手動再起動
- CDN/Workerの直接デプロイ（ビルドレスの変更）
- 設定ファイルのみの変更で挙動を切り替えられる設計

## 5. 新機能は本番データに影響しない場所でテストする

**原則**: 本番のデータディレクトリ、本番のCDN、本番のDBに直接書き込む新機能は、まずステージング環境またはdryrunモードで検証する。

**具体例**:
- 新しいファイル生成機能: まずコンソール出力のみで動作確認
- 新しいスケジュールジョブ: まず手動実行で結果を確認
- 新しいCDN同期: まずローカルのみで書き出し、問題ないことを確認後にCDN同期を有効化

## 6. ロールバック可能な変更を心がける

**原則**: 変更は可逆的に行う。不可逆な操作（ファイル上書き、DB変更、CDNアップロード）の前に、元の状態に戻せる手段を確認する。

**対策**:
- バックアップの定期実行と復元手順の確認
- デプロイ前のスナップショット（git tag、DB dump）
- CDNにアップロードしたものを削除・差し替えできるツールの整備
