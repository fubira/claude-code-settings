# 問題解決の原則：本質を見極める

**Category**: Problem Solving
**Tech Stack**: Universal
**Applicability**: Universal
**Added**: 2025-01-19

## 核心原則

**問題の本質を見極めずに対症療法を繰り返さない**

## 判断基準：本質的な問題か、対症療法か

### ✅ 本質的なアプローチ
- 問題が起きる根本原因を特定する
- 「なぜ動かないのか」を理解してから修正する
- 動作している部分は変更しない

### ❌ 対症療法（避けるべき）
- エラーが出たら別の方法を試す
- 原因不明のまま回避策を重ねる
- 動作している実装を書き換える

## 手詰まり時の対処

### 1. 目線を広げる

**問いを変える**：
- 「どうやって直すか」→「何が原因か」
- 「この実装で」→「このレイヤーで正しいか」
- 「コードを変える」→「設定・環境・構成を確認する」

**スコープを広げる**：
- コード → ビルド設定 → 環境変数 → プラットフォーム固有の問題

### 2. 既知の問題を調査する

**必ず調べる場所**：
1. 公式ドキュメントのトラブルシューティング
2. GitHub Issues（ライブラリ・フレームワーク）
3. Stack Overflow
4. Web検索（エラーメッセージ全文）

**効果的な検索**：
- エラーメッセージの核心部分を使う
- ツール名 + 環境 + 問題の性質
- 「動く環境」と「動かない環境」の違いをキーワードに

### 3. 同じ問題を見つけたら

- Issue/解決策を複数読む（1つだけで判断しない）
- 解決策の「理由」を理解する
- 自分の環境に適用可能か検証する

## 実装変更の判断

実装を変更するのは、以下を全て確認した後：

- [ ] 動作環境と非動作環境の差異を特定した
- [ ] 公式ドキュメント・Issuesを調査した
- [ ] 設定・環境変数・ビルド構成を確認した
- [ ] 根本原因を理解した

## まとめ

1. **本質を見極める** - 対症療法に走らない
2. **目線を広げる** - 視野を狭めない
3. **既知の問題を調べる** - 先人の知恵を活用する
