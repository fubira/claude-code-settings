## 作業方針

### コーディング

- **読んでいないファイルを編集しない**。必ず Read してから Edit/Write する
- 既存のコード規約・パターンに従う
- 関数型・宣言型優先、DRY、早期リターン
- 引数・戻り値は構造化（TS: ROROパターン）
- I/O並列化（TS: `Promise.all`、Go: goroutine）
- 頻出定数はモジュールレベルで定義

### 環境依存の排除

- **ローカルパス・IPアドレスをコードに書かない**。環境変数か設定ファイルで外部化する
- パス解決は `$(dirname "$0")/..`、`import.meta.dirname`、`Path(__file__)` 等の相対解決
- テストフィクスチャにも実在パスを使わない（`/tmp/test-dir` 等を使う）

### 品質

- コード変更後はリント・テスト実行
- `package.json`のscriptsを確認してから実行
- テストはCo-location（ソースと同じディレクトリ）
- エラー処理・エッジケース優先、早期リターン
- リファクタリングと機能追加は別コミット

### Bash ツール

- `echo "---"` 等の見た目用の区切りコマンドを入れない（権限確認が発生して止まるため）
- 複数情報の確認は並列ツール呼び出しか `&&` で対応
- **インラインコードよりファイル優先**: `python -c '...'` やヒアドキュメントでのインライン実行は記録に残らず、毎回権限確認も挟まる。実験・検証・テスト用コードは原則ファイルに書き出してから実行する。インラインは本当に使い捨ての確認にのみ使う
- **サーバーでの確認はシンプルに**: `cat`/`grep`/`head` 等を優先
  - Pythonワンライナーやスクリプトのコピペはターミナルで改行・スペースが崩れるため最後の手段
- **`| tail -N` をバックグラウンドや長時間コマンドに使わない**: `tail -N` はストリーム全体を読み終えてから最後のN行を出力する。完了まで一切出力が見えなくなる。代わりにログファイルにリダイレクト（`>> /tmp/log.txt 2>&1`）して `tail -f` で監視する

### 問題解決・デバッグ

本質を見極め、対症療法に走らない。

- **根本原因を理解してから実装変更**。動作/非動作環境の差異を特定（コードでなく設定・環境の問題かも）
- **手詰まり時は視野を広げる**（コード → ビルド → 環境変数 → プラットフォーム）。公式ドキュメント・Issues・SO も調べる
- **原因が見つかるまでやり抜く**。「たまたま」「シードのせい」「運」で片付けない。再現性のある現象には構造的理由がある
- **外部のせいにする前に自分のコードとログ**。バージョン違い・ライブラリバグを憶測で挙げない
- **時間コストに不寛容に**。長時間テストを安易に提案せず、まず小さく速く確認

## テスト

- カバレッジ80%以上目標
- 実装を先に読んでからテストを書く
- 境界値（±1）を必ずテスト
- 確率的処理は統計的に検証
- テスト失敗時は期待値を疑う
- 自作コードのみテスト（外部ライブラリの動作はテストしない）

## CLAUDE.md とメモリの管理ポリシー

### CLAUDE.md（プロジェクト）

**間違えやすい情報・毎回必要な情報**を書く場所。毎会話読み込まれる。

| 書く | 書かない |
|------|---------|
| CLI リファレンス（コマンド名、引数、排他制約、デフォルト値） | コードから読み取れる実装詳細 |
| 操作ルール（フィードバックから昇格した禁止・推奨） | 時系列の経緯・意思決定の過程（→メモリ/ジャーナル） |
| やらなくてよいことリスト（不採用アプローチ） | 変動する数値（→メモリ） |
| アーキテクチャの制約 | |

### メモリ

会話をまたぐ補足コンテキスト。MEMORY.md は全行システムプロンプトに載るため**量が品質に直結する**。

| 書く | 書かない |
|------|---------|
| 現在の戦略パラメータ・運用状態（変動するもの） | CLAUDE.md に統合済みのルール |
| 外部リソースの参照先（API、サーバ環境） | 撤退済み戦略の詳細（→CLAUDE.md に1行で統合） |
| CLAUDE.md に書くと長すぎるフィードバックの詳細背景 | コード・git から読み取れる情報 |
| 進行中タスク・判断待ち事項 | |

### 昇格パターン: メモリ → CLAUDE.md

フィードバックが「同じカテゴリの新しい間違いを防げない」なら、個別記録ではなくルールとして CLAUDE.md に昇格させメモリファイルは削除する。

### MEMORY.md の予算

**80行以下**を維持。超えたら CLAUDE.md に昇格・撤退済み項目を1行に圧縮・個別メモリファイルを統合する。
